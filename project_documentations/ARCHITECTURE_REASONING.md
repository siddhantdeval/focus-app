# Architecture Logic & Technical Trade-offs

## Chosen Architecture: Rust Core + Native UIs (SwiftUI, WinUI, Compose)

### Technical Details
*   **Core Domain (Rust):** Contains all business logic, timers, recurrence engines, JSON serialization/deserialization, state reducers, and data persistence layers.
*   **FFI Layer (UniFFI):** A Mozilla-backed tool that mathematically generates unsafe C-ABI boundaries between Rust and host languages (Swift, Kotlin, C#). It creates idiomatic, memory-safe wrappers on the native side.
*   **Database (rusqlite):** SQLite engine embedded directly within the Rust core. The native UIs never speak to SQL. They subscribe to Rust channels/streams and receive parsed UI-ready structs.
*   **Native GUI:** Completely decouples presentation from logic. A SwiftUI `View` only contains Swift layout code and binds to a Swift class generated by UniFFI.

### Trade-offs of Chosen Architecture
*   **Pros:** Absolute maximum performance, indistinguishable native feel on every OS, mathematically safe memory management (no crashes), identical business logic across platforms, battery efficient.
*   **Cons:** Extremely steep learning curve (Rust + FFI concepts), highest initial development time, debugging across language boundaries (Rust to Swift) requires sophisticated tooling (LLDB), requires developers to know layout systems for 3 different OSs.

## Alternative 1: Electron / WebWrappers (Tauri, React Native for Desktop)

### Technical Details
*   Uses HTML/CSS/JS deployed via a bundled Chromium browser instance (Electron) or OS WebKit engine (Tauri).
*   Business logic is entirely written in JavaScript/TypeScript running inside a Node.js context.

### Trade-offs
*   **Pros:** Fastest time-to-market. A single React/Vue codebase builds macOS, Windows, iOS, and Android simultaneously. Easiest to hire for (web developers). 
*   **Cons:** 
    *   **Resource Bloat:** A blank Electron app idles at ~150MB of RAM. A native macOS timer idles at ~15MB.
    *   **Battery Drain:** Chromium's rendering pipeline forces the GPU and CPU to wake constantly, draining laptop batteries rapidly.
    *   **Uncanny Valley:** Right-click context menus, scroll physics, accessibility (VoiceOver/Narrator), and text-selection never feel truly native.
    *   **Background Restrictions:** iOS and Android aggressively kill JS background engines. Relying on web views for a precise Pomodoro timer on mobile is highly unreliable.

## Alternative 2: Flutter / Dart

### Technical Details
*   Google's UI framework that bypasses native controls entirely. It draws every single pixel on the screen itself using the Skia (or Impeller) C++ graphics engine natively.
*   Dart compiles ahead-of-time (AOT) to native ARM/x86 code.

### Trade-offs
*   **Pros:** High performance (60/120fps), single UI codebase for all platforms (mobile, desktop, web), extremely fast development cycle with Stateful Hot Reload.
*   **Cons:**
    *   **Non-Native Look:** Because Flutter draws its own UI, it does not use the underlying OS buttons, text fields, or menus. You have to emulate Cupertino (macOS/iOS) or Fluent (Windows) designs. 
    *   **Accessibility Hurdles:** Screen readers have to translate Flutter's custom rendering tree back to the OS, which is often buggy.
    *   **Platform Integrations:** Hooking into native OS features (like macOS Menu Bar tray icons or Windows Notifications) requires writing custom native MethodChannels in Swift/Kotlin/C# anyway, defeating the purpose of a cross-platform framework for highly-integrated OS features.

## Alternative 3: Kotlin Multiplatform (KMP)

### Technical Details
*   JetBrains' ecosystem where business logic is written in pure Kotlin.
*   Compiles Kotlin to Java Bytecode (Android), Objective-C/Swift Native Binaries via LLVM (Apple), and native executables (Windows).
*   UIs are written natively (SwiftUI, Compose, WinUI).

### Trade-offs
*   **Pros:** Native UIs (like our Rust approach), but with a significantly lower learning curve than Rust (Kotlin is basically the modern Java). Excellent tooling via Android Studio/IntelliJ.
*   **Cons:**
    *   **iOS Interop Tax:** Kotlin compiles to Objective-C headers (not Swift natively), which makes consuming KMP libraries in modern SwiftUI feel clunky and requires wrapper classes.
    *   **Garbage Collection:** Kotlin Native introduces its own memory management model (Garbage Collector) to iOS natively, which can cause micro-stutters resulting in dropped frames during intensive data processing. Rust (Zero-Cost abstractions) has no garbage collector.
